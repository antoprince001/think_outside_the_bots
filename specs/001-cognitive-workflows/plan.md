# Implementation Plan: Cognitive Learning Workflows

**Branch**: `001-cognitive-workflows` | **Date**: 2026-07-20 | **Spec**:
[spec.md](./spec.md)

**Input**: Feature specification from `/specs/001-cognitive-workflows/spec.md`

## Summary

Turn the existing visual React/Vite starter into a local-first learning application that makes
students contribute reasoning before AI help becomes available. Retain the current React, Vite,
and lucide-react framework choices. Implement a declarative workflow interpreter shared by the
Feynman, Socratic, AI-freeze, long-draft, and custom workflows; persist learning work locally;
and call only the selected provider directly from the browser. API keys remain in browser-local,
session-scoped storage and are never sent to application-owned services.

## Technical Context

**Language/Version**: JavaScript ES modules; React 19.2.7 and React DOM 19.2.7

**Primary Dependencies**: Vite 8.1.5, @vitejs/plugin-react, lucide-react; add Vitest, jsdom,
React Testing Library, and user-event as development-only test tooling.

**Storage**: Browser-local storage only. Persist non-secret preferences, workflows, tasks, drafts,
and session records locally. Keep API keys only in memory/sessionStorage for the active browser
session; do not place keys in localStorage, URLs, exports, logs, analytics, error reports, or an
application backend.

**Testing**: Vitest for workflow/state/storage units and React Testing Library for user flows;
mock provider requests. Run the production Vite build as the initial performance check.

**Target Platform**: Modern desktop and mobile browsers supported by the current Vite application.

**Project Type**: Client-side single-page web application.

**Performance Goals**: Local gate, countdown, draft-progress, setup, selection, save, and review
actions complete within 1 second under normal use; at 100 active sessions, 95% of these local
actions complete within 2 seconds. Provider response time is measured separately and excluded.

**Constraints**: Preserve the existing framework choices; no application backend; support only
provider adapters that permit secure browser-origin requests; protect the key from display and
from application-controlled transmission; maintain accessible responsive workflows.

**Scale/Scope**: First release serves one student per browser profile with four prebuilt workflows,
limited-step custom workflows, direct supported-model connections, and locally retained sessions.
Classrooms, sharing, arbitrary workflow code, and server synchronization are out of scope.

`src/main.jsx` into presentation components, pure workflow logic, and narrow browser services.
This preserves the current framework while making workflow rules independently testable.
